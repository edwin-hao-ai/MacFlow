## 一、产品概述

### 1.1 产品定位
MacSlim 是 **Mac 端专属、规则驱动、极简一键式的系统运维工具**，同时服务开发者和普通用户。核心解决四大高频痛点：

- 开发者：进程冗余卡顿、NPM/Docker/Xcode 缓存占满硬盘、端口被占、残留进程
- 普通用户：软件关闭残留、后台冗余进程、系统卡顿、硬盘变慢

主打「打开就扫、规则自动分类、一键全量优化、全透明可审计、安全边界明确」。

**刻意不做 AI**：所有分类和风险判断用规则引擎。理由：规则足够解决 95% 场景；AI API 调用会毁掉低价订阅定价；用户更信任可审计的规则而非黑盒推理。

### 1.2 核心价值主张
- 比活动监视器简单：零命令零配置，一键完成扫描 + 清理
- 比 CleanMyMac 轻量 + 透明：安装包 <15MB（vs CleanMyMac 500MB+），无广告无捆绑，每步操作可审计
- 比所有工具都懂开发者：深度适配 NPM/Docker/Xcode/Homebrew 开发场景，自动识别正在运行的服务不误关
- 视觉质感对标 CleanMyMac：普通用户敢用、开发者不会觉得 low

### 1.3 核心差异点（与所有竞品的本质区别）
| 竞品 | MacSlim 核心优势 |
| :--- | :--- |
| CleanMyMac | 极致轻量、开发者场景深度适配、价格仅 1/10、全透明不黑盒、CLI 支持 |
| App Tamer | 不仅能暂停进程，还能清理残留/重复进程、释放端口、清理开发缓存、一键全量优化 |
| Memory Clean | 从根源清理冗余进程，而不是粗暴清内存，效果持久不反弹 |
| 活动监视器 | AI 自动分类、一键批量操作、安全白名单、回滚机制、开发场景专属适配 |

### 1.4 非目标（绝对不做的功能）
- 不做 Windows/Linux 版本（MVP 阶段仅支持 Mac）
- 不做杀毒、防火墙、隐私保护等安全功能
- 不做应用卸载、大文件扫描、重复文件清理等通用功能
- 不做任何广告、推送、数据收集（所有数据本地存储）
- 不接 AI / LLM API

## 二、技术栈（Vibe Coding 可搜索验证后优化）

[SEARCH] 2026 年 Tauri v2 最新稳定版特性及最佳实践
[SEARCH] macOS Sequoia 系统权限及 API 变更

```
- 桌面端框架：Tauri v2（禁止 Electron）
- 前端：SolidJS + TypeScript + Tailwind CSS（禁止 React / Vue / 重型 UI 组件库）
- 动画库：motion-one（3KB）按需引入，或原生 CSS transition
- 图标：lucide-solid 按需 tree-shake
- 后端：Rust（Tauri 原生命令 + IPC）
- CLI：Rust 编写，与桌面端共享 core crate
- 系统 API：sysinfo + nix + bollard（Docker）+ rusqlite（本地存储）

- 不做：AI / LLM API、任何需要网络的功能（软件更新除外）
```

**技术栈决策理由（2026-04-21）**：
- 选 Solid 不选 React：bundle 小 3-5 倍，无 Virtual DOM 开销，Tauri + Solid + Tailwind 组合在 Spacedrive 等产品已验证
- 删除 AI：规则引擎够用；API 成本毁定价；用户更信任可审计规则

### 3.2 进程优化模块（优先级：最高）

#### 3.2.1 功能描述
自动扫描所有系统进程，按安全等级+场景分类，自动过滤系统核心进程和前台活跃进程，一键清理冗余/残留/闲置进程，释放 CPU/内存/端口。

#### 3.2.2 进程分类规则（严格遵循）
| 安全等级 | 分类名称 | 识别规则 | 默认选中 |
| :--- | :--- | :--- | :--- |
| [*] 绝对安全 | 软件关闭残留进程 | 父进程已退出、无前台窗口、无 IO 操作、闲置 >5 分钟的僵尸/孤儿进程 | 是 |
| [*] 绝对安全 | 重复冗余进程 | 同一可执行文件、同一父进程、无前台窗口、闲置 >5 分钟的重复进程 | 是 |
| [!] 低风险 | 长期闲置后台进程 | 无前台窗口、闲置 >30 分钟、非系统核心、非白名单、资源占用 >5% | 是 |
| [!] 低风险 | 高占用闲置进程 | 无前台窗口、闲置 >10 分钟、CPU>20%/内存 >1GB/高能耗 | 是 |
| [D] 开发者专属 | 开发环境闲置进程 | Node/Python/Java/Docker 闲置进程、僵死进程、占端口的残留进程 | 是 |
| [x] 不推荐 | 系统核心进程 | macOS SIP 保护进程、系统核心服务、内核进程 | 否（隐藏） |
| [x] 不推荐 | 前台活跃进程 | 5 分钟内有操作的前台窗口、正在运行的软件主进程 | 否（隐藏） |

[SEARCH] 2026 年 macOS Sequoia 系统核心进程完整白名单
[SEARCH] Apple Silicon 与 Intel 芯片进程差异
[THINK] 如何精准区分正在运行的 Node 开发服务和闲置残留进程

#### 3.2.3 核心功能
1. 自动端口占用检测：识别被进程占用的端口，标注在开发残留进程中
2. 优雅关闭优先：优先发送 SIGTERM 信号，只有进程无响应时才发送 SIGKILL
3. 进程白名单：支持手动添加进程到白名单，永久不扫描不清理
4. 一键回滚：清理前自动创建进程快照，清理后 10 分钟内可一键恢复所有进程

#### 3.2.4 技术实现要点
- 进程信号使用 Rust `nix::sys::signal::kill`（即 POSIX `kill(2)` 系统调用，就是原生 API；原 PRD 的「禁止 kill 命令」约束作废 2026-04-21）
- 优先 SIGTERM，3 秒无响应再 SIGKILL；所有操作记入本地 SQLite 日志
- 系统核心进程白名单内置（见 CLAUDE.md §4.4），每次启动校验更新
- 进程列表通过 `sysinfo` crate 获取，不走 ps/lsof shell
- 端口检测用 `netstat` 原生系统调用或读 `/dev/lsof` 的替代 API；仅在无 native 方案时回退到 `lsof` 子进程
- **不承诺进程回滚**：被终止的进程内存状态丢失，只能记录进程元数据（可执行路径、参数），供用户手动重启参考

### 3.6 安全与回滚模块（优先级：最高）

#### 3.6.1 6 重安全防护机制
1. **系统核心白名单**：内置 macOS SIP 保护目录、系统核心进程白名单，绝对不触碰
2. **风险分级机制**：只有 100% 无风险的内容才会默认选中，低风险内容默认不选，高风险内容默认隐藏
3. **原生命令优先**：所有清理操作优先使用工具原生命令，绝不手动删除文件
4. **预执行校验**：所有操作执行前先模拟执行，验证无风险后再真实执行
5. **一键回滚机制**：所有操作执行前自动备份元数据，10 分钟内可一键回滚
6. **增量扫描**：只扫描固定的缓存路径，绝不扫描用户的文档、照片、工作目录

#### 3.6.2 操作日志与恢复指引（替代原「一键回滚」）

> **2026-04-21 修正**：Docker/NPM/Xcode 缓存清理本质不可逆，进程 SIGKILL 后内存状态无法恢复。原「一键回滚」表述作废，改为「操作日志 + 恢复指引」，避免过度承诺。

- **操作日志**：每次清理生成 SQLite 记录，包含时间、项目列表、释放空间、被终止进程的 path/args。用户可在「历史」面板查看
- **可真正撤销的项**（移动到 Trash，保留 24 小时）：白名单变更、设置项、`~/Library/Caches/xxx` 中用户级缓存的元数据
- **不可撤销项**（UI 必须在清理前明确告知）：
  - `npm/yarn/pnpm cache clean`
  - `docker image/builder/container/volume prune`
  - `brew cleanup`
  - Xcode DerivedData 删除
  - 进程 SIGKILL

### 3.8 应用程序管理模块（新增 2026-04-22）

#### 3.8.1 功能描述
macOS 的进程和应用是分离的：Chrome「一个应用」对应 20+ 个进程。
本模块按 `.app` bundle 聚合进程，展示用户视角的「运行中的应用」，
解决活动监视器里 Chrome Helper 刷屏看不懂是哪个应用的痛点。

#### 3.8.2 核心能力
- 按 bundle path 分组进程（main + 所有 Helper）
- 每个应用卡片显示：图标占位 / 名称 / Bundle ID / 总内存 / 总 CPU /
  进程数 / 运行时长 / 监听端口
- 读 Info.plist 拿 `CFBundleDisplayName` / `CFBundleIdentifier`
- 两个操作：
  - **退出**：`osascript -e 'tell application "X" to quit'`
    触发 macOS 标准退出流程（保存未存文件提示等）
  - **强制退出**：对所有相关 PID 调 graceful_kill（走进程树清理）
- 筛选：搜索 + 「隐藏系统应用」开关（默认开）
- 5 秒自动刷新

#### 3.8.3 技术实现
- `src-tauri/src/applications.rs`：AppInfo 结构 + list_running_apps
- 用 sysinfo 扫进程 → 按 `find_app_bundle` 提取 `.app` 路径分组
- Info.plist 仅解析 XML 格式（bplist 不解析，回退到路径推断）

### 3.9 Docker 深度视图（新增 2026-04-22）

#### 3.9.1 功能描述
独立于缓存清理的批量模式，Docker 视图提供**细粒度的镜像/容器/卷管理**，
像 Docker Desktop 自带界面一样但更轻。

#### 3.9.2 核心能力
顶部概览栏：可回收大小 / 镜像数（含悬空数） / 容器数（含停止数） /
卷数（含未引用数） / 构建缓存总大小 / 「一键回收所有」按钮。

三个 tab：
- **镜像**：id / repo:tag / 大小 / 创建时间 / 悬空标记 / 使用中标记 / 删除按钮
- **容器**：id / name / image / 状态 / 运行中/已停止 / RW 层大小 / 删除按钮
- **卷**：name / driver / 使用中标记 / 删除按钮（仅未使用可删）

操作：
- 单项删除：`docker image rm -f ID` / `docker rm -f ID` / `docker volume rm -f NAME`
- 一键回收：`docker system prune -f --volumes`（含二次确认对话框）

未运行时显示「Docker 未运行或未安装」。

#### 3.9.3 技术实现
- `src-tauri/src/docker.rs`：统一经 `docker` CLI，结构化 Go template 输出
- 独立探测 daemon：`docker info --format {{.ServerVersion}}`
- 容器使用中判定：ps 输出 Image 列交集
- 卷使用中判定：解析 ps --format {{.Mounts}}
- 镜像 in_use：比对容器的 image tag 和 digest

# 八、Office Hours 评审（2026-04-21，Garry Tan 视角）

这份 PRD 很有激情，ASCII 美学的产品直觉也很真诚。但作为「要真的 ship 并被付费用户使用」的文档，它有几个**会把产品带偏的前提错误**，需要在写代码之前处理掉。

## 8.1 需要推翻或修正的前提（Premise Challenge）

### [BLOCK-1] 「10MB 安装包 + <1% CPU + <50MB 内存」是三选一，不是三选三
- Tauri v2 hello-world 实测 2-3MB，realistic app 2-10MB（2026 年数据）。ASCII-only UI 确实省，但你要内置：进程扫描引擎、Docker/NPM 子进程管理、历史回滚快照、系统托盘图标、签名 + notarization。**真实估算：ARM64+x86_64 通用二进制 15-20MB**。
- 「后台常驻 CPU <1%」与 3.5.4「托盘状态刷新 1 秒/次」直接冲突。每秒轮询 CPU/内存/磁盘 + 进程列表，在 M 系列上实测约 2-4% CPU。
- **修正**：改成「安装包 <15MB（Universal 2）」、「后台空闲 CPU <2%」、「内存 <80MB」。给自己留余量，别把不现实的数字写成验收标准。

### [BLOCK-2] 「禁止使用 kill 命令行工具，必须使用 macOS 原生 API」这条是反的
- `kill(1)` 是 `kill(2)` syscall 的薄包装，**它就是原生 API**。
- macOS 真正的「原生进程控制」需要 `task_for_pid()` + Debugging Tool Entitlement，而这个 entitlement 需要 Apple 特批，普通开发者证书拿不到。非 SIP 保护进程用 `libc::kill(pid, SIGTERM)` 就是最干净的做法。
- **修正**：直接用 `nix::sys::signal::kill` crate（Rust 原生封装），先 SIGTERM 等 3 秒，再 SIGKILL。删掉 3.2.4 那条误导性约束。

### [BLOCK-3] 「绝对安全不误删」+「一键回滚」是过度承诺
- `docker image prune -f`、`npm cache clean --force`、`brew cleanup` 这些命令**本质上不可逆**。回滚「提示用户重新拉取/安装」不是回滚，是认错。
- 进程快照回滚更难：被 SIGKILL 的进程丢了内存状态，重启 ≠ 恢复。用户的 Chrome 标签页、未保存文件、开发服务的 in-memory 状态全没了。
- **修正**：
  - 把「一键回滚」降级为「**操作日志 + 重新获取指引**」，明说「文件清理不可逆，进程清理后需重启软件」。
  - 真正可回滚的只有「白名单变更」「设置变更」「被移动到 `~/.Trash/macslim-YYYYMMDD/` 的元数据」。
  - 默认选中的边界从「100% 安全」收紧到「重新获取成本 < 5 分钟」。

### [BLOCK-4] 「AI 驱动」在 MVP 里是空概念 ✅ 已决策：砍掉
- 2026-04-21 用户决策：删除所有 AI / LLM 相关内容，MVP 和 v1 纯规则驱动
- 理由：API 成本毁定价、增加隐私顾虑、规则足够

### [BLOCK-5] 纯 ASCII UI ✅ 已决策：放弃 ASCII，改 CleanMyMac 风
- 2026-04-21 用户决策：UI 视觉对标 CleanMyMac，放弃 ASCII 美学
- 新规范：见 CLAUDE.md §3
- 原 PRD §3.1.3、§四全部作废

### [WARN-1] 「比 CleanMyMac 轻量 100 倍，价格 1/10」是把竞品看扁了
- CleanMyMac 用户留下来不是因为功能，是因为**信任感**和**「清理了多少 GB」的视觉满足感**。纯 ASCII 没有「进度条填满那一刻的多巴胺」。
- 「开发者场景深度适配」是真差异点，但「比 CleanMyMac 简单 100 倍」对开发者不重要 —— 他们本来就会用命令行。你的**真实竞品是 `rm -rf node_modules && docker system prune -a`**。
- **修正定位**：从「比 CleanMyMac 简单」转向「**把 5 条我每次都要查的清理命令，变成 1 个命令**」。你的用户是「知道要清理但懒得每次 Google 命令」的开发者，不是「不懂技术」的用户。

### [WARN-2] 「免费版每天限 3 次清理」的定价模型违反用户直觉
- 清理工具不是 SaaS。用户一个月可能只用 2 次，限次数没意义；或者连续 debug 时一天用 10 次，3 次锁喉会激怒用户。
- **修正**：Pro 功能不是「次数」，是「**自动监控 + 定时清理 + 多设备白名单同步 + Docker 3 个月未用镜像清理**」这类「持续价值」功能。免费版无限次手动清理 + 基础规则，Pro 买的是「不用手动触发」。

### [WARN-3] MVP 4 周里程碑严重低估
- 里程碑 1 的「2 周完成核心 + 托盘 + CLI」对单人开发者是 **4-6 周**的工作量，考虑到：
  - Apple notarization 首次配置通常 1-2 天踩坑
  - Universal 2 打包 + 代码签名的 CI
  - 测试覆盖你的安全承诺需要大量 fixture
- **修正**：总周期从 4 周改为 **8-10 周**，或者砍功能 —— 删掉里程碑 1 的 CLI 和托盘，挪到里程碑 3。
